  12  This module contains the base GraphElement, Node, Edge, 
  13  and Graph implementations for Graphine, an easy-to-use, 
  14  easy-to-extend Graph library. 
  15   
  16  Graphs generated using this representation can have  
  17  directed or undirected edges, the ability to represent  
  18  loops and parallel edges, and can attach any hashable data  
  19  to nodes and edges. 
  20   
  21  Interface summary 
  22  ================= 
  23   
  24  To create a new Graph: 
  25   
  26          >>> from graph.base import Graph 
  27          >>> g = Graph() 
  28   
  29  To add nodes: 
  30   
  31          >>> node_1 = g.add_node(name="bob") 
  32          >>> node_2 = g.add_node(weight=5) 
  33          >>> node_3 = g.add_node(color="red", visited=False) 
  34          >>> node_4 = g.add_node() 
  35   
  36  To add edges: 
  37   
  38          >>> edge_1 = g.add_edge(node_2, node_3) 
  39           
  40  Notice that edges can have properties as well: 
  41   
  42          >>> edge_2 = g.add_edge(node_1, node_2, weight=5) 
  43          >>> edge_3 = g.add_edge(node_3, node_1, stuff="things") 
  44

  46  keyword argument, which, when set to False, will cause the edge 
  47  created to be bidirectional. Its default value is True, ie, 
  48  edges are by default directed. 
  49

  58          >>> g.remove_edge(edge_4) 
  59

  61  remove that edge as well. 
  62   
  63  Navigating Graphs 
  64  ----------------- 
  65   
  66  In addition to storing your data, Nodes and Edges have 
  67  a few other special properties by default. 
  68   
  69  For Nodes, these properties are "incoming" and "outgoing",  
  70  and they contain the incoming and outgoing edges attached 
  71  to that node. The "bidirectional" property is also there 
  72  in case you only want the edges which go both ways. An 
  73  additional property "edges" contains all of these elements. 
  74   
  75  For Edges, these properties are (again, by default)  
  76  "start" and "end". Because edges can be bidirectional, 
  77  they also have an "is_directed" property, and provide 
  78  the convenience function "other_end", which takes a 
  79  node and, if possible, returns the opposite endpoint 
  80  incident to that edge. 
  81   
  82  The "key" attribute is also provided for use in 
  83  dictionaries and other settings where the essential 
  84  question is whether two graph elements are equivalent, 
  85  not whether they are the exact same element. 
  86   
  87  To get all the outgoing edges of a particular node: 
  88   
  89          >>> n1.outgoing 
  90          [Edge(weight=5)] 
  91   
  92  And to get the incoming edges: 
  93   
  94          >>> n2.incoming 
  95          [Edge(weight=5)] 
  96   
  97  The same sorts of things work for the properties 
  98  "bidirectional" and "edges". 
  99   
 100  You can use this to get all the nodes adjacent to 
 101  an interesting Node: 
 102   
 103          >>> adjacent = [edge.end for edge in node_1.outgoing] 
 104   
 105  To get the degree of the given node: 
 106   
 107          >>> degree = node1.degree 
 108           
 109  You can combine this with Edges' "start" and "end" properties 
 110  to navigate your graph, if you know how your data is related 
 111  structurally. For example, to get the oldest outgoing edge attached 
 112  to the oldest node attached to node_1, I could do the following: 
 113   
 114          >>> interesting_node = node_1.outgoing[0].end.outgoing[0] 
 115   
 116  In addition to the "start" and "end" properties, you can use the  
 117  "other_end" function, which takes a node and, if possible, returns 
 118  the edge's other end. If it cannot follow the edge to a point  
 119  opposite the given node- for instance, because the edge is directed 
 120  the other way- it raises AttributeError. 
 121   
 122          >>> endpoint = edge_1.other_end(node_2) 
 123          >>> endpoint == node_3 
 124          True 
 125          >>> edge_1.other_end(node_3) 
 126          ... 
 127          AttributeError: Edge() contains no endpoint opposite to Node() 
 128   
 129  You will frequently see "other_end" used in situations where it is 
 130  possible that the given edge could be either directed or undirected, 
 131  or where the orientation of a directed edge is unknown. 
 132   
 133  Of course, there are scenarios in which you don't know how to 
 134  navigate your graph, or need to find a starting point that isn't 
 135  easily determined based on a given element's properties. For that, 
 136  Graph supports Node and Edge iteration for the simple cases, and 
 137  traversals and walks for more complex behavior. 
 138   
 139  To iterate over all the nodes in a graph: 
 140   
 141          >>> for node in g.nodes: 
 142          >>>     print(node) 
 143          Node(name="bob") 
 144          Node(weight=5) 
 145   
 146  To do the same for edges: 
 147   
 148          >>> for edge in g.edges: 
 149          >>>     print(edge) 
 150          Edge() 
 151          Edge(weight=5) 
 152   
 153  If you only want certain nodes, the search 
 154  functions are provided for convenience: 
 155   
 156          >>> for node in g.search_nodes(name="bob"): 
 157          >>>     print(node) 
 158          Node(name="bob") 
 159           
 160  And for edges: 
 161   
 162          >>> for edge in g.search_edges(weight=5): 
 163          >>>     print(edge) 
 164          Edge(weight=5) 
 165   
 166  You can also get the elements which are equivalent to a 
 167  given one, for example to compare elements between graphs. 
 168  This is done using the "key" attribute and dictionary-style 
 169  access. 
 170   
 171          >>> g[node_1.key] 
 172          Node(weight=5) 
 173   
 174  This technique also works for edges. 
 175   
 176  In addition to the datawise and unordered views of graphs, 
 177  several methods for ordering based on structural properties 
 178  are provided. The most important of these are traversals 
 179  and walks. 
 180   
 181  Three traversals are provided by default- A*, depth first, 
 182  and breadth first. They are all guarenteed to only visit 
 183  a given node once, and visit nodes in a different order 
 184  based on how they are related structurally. 
 185   
 186  To do a depth first traversal: 
 187   
 188          >>> for node in g.depth_first_traversal(node_1): 
 189          >>>     print(node) 
 190          Node(name="bob") 
 191          Node(weight=5) 
 192   
 193  Breadth first traversals are similar in usage, simply 
 194  substituting "depth_first_traversal" for "breadth_first_traversal". 
 195   
 196  A* traversals require an additional callable argument. 
 197  This callable will be passed a list of unvisited nodes 
 198  and should select and remove exactly one of them to return. 
 199   
 200  For example, to navigate a graph by visiting the most 
 201  popular (ie, most incoming edges) nodes first, I could 
 202  write a getter function as follows: 
 203   
 204          >>> def get_popularity(node): 
 205          >>>     return len(node.incoming) 
 206   
 207  A selector function: 
 208   
 209          >>> def get_most_popular(nodes): 
 210          >>>     nodes.sort(nodes, key=get_popularity) 
 211          >>>     return nodes.pop() 
 212   
 213  And traverse the graph: 
 214   
 215          >>> for node in g.a_star_traversal(node_1, get_most_popular): 
 216          >>>     print(node) 
 217          Node(weight=5) 
 218          Node(name="bob") 
 219   
 220  The walks, in contrast to the traversals, can visit a node 
 221  or edge more than once. They are the most powerful structural 
 222  technique provided for navigating graphs, but are also probably 
 223  the hardest to use. 
 224   
 225  The walks are implemented as generators, with each step yielding 
 226  a set of nodes (for walk_nodes) or edges (for walk_edges) that 
 227  are potentially "next" in terms of adjacency. The application 
 228  should select one of those and use send() to send it back. 
 229   
 230  The trick is that generators must be fed None in their first 
 231  send(), which can complicate application logic. For example: 
 232   
 233          >>> w = g.walk_nodes() 
 234          >>> w.send(None) 
 235   
 236  Now we send it the node we're interested in walking away from: 
 237   
 238          >>> candidates = w.send(node_1) 
 239          >>> candidates 
 240          {Node(weight=5)} 
 241   
 242  We get back the set of nodes to which we can walk. We should 
 243  select one and send it back: 
 244   
 245          >>> w.send(candidates.pop()) 
 246          set() 
 247   
 248  To end the generator's run: 
 249   
 250          >>> w.send(None) 
 251          ... 
 252          StopIteration 
 253   
 254  walk_edges operates identically but accepts edges and yields 
 255  sets of edges. 
 256           
 257  Binary Graph Operations 
 258  ----------------------- 
 259   
 260  Four basic operations are provided for the comparison 
 261  of graphs:  
 262   
 263          1. union (|), which creates a new graph containing all 
 264             the nodes and edges in either of its parents, 
 265   
 266          2. intersection (&), which creates a new graph containing 
 267             all the nodes and edges in both of its parents, 
 268   
 269          3. difference (-), which creates a new graph containing 
 270             all the nodes and edges in the first parent but 
 271             not in the second. 
 272   
 273          4. merge (+), which creates a new graph with all the  
 274             data-unique nodes from both parent graphs plus all  
 275             their structurally and data unique edges. 
 276   
 277  For more information on the usage of these functions, consult their 
 278  individual docstrings.
